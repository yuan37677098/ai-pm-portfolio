# 飞小书 · AI 智能客服助手

基于 RAG（检索增强生成）技术搭建的飞书智能问答助手，端到端从 0 到 1 独立完成。

## Demo 体验

本地运行：`python app.py`，浏览器打开 http://localhost:7860

## 技术架构

用户提问 → Embedding 向量检索 → 召回 Top 5 相关文档块 → LLM 生成回答 → 流式输出

| 组件 | 技术选型 | 选型理由 |
|------|----------|----------|
| LLM | DeepSeek-Chat | 中文能力强，性价比高 |
| Embedding | all-MiniLM-L6-v2 | 本地免费，无需 API 费用 |
| 向量库 | ChromaDB | 轻量，本地部署，Python 原生 |
| 前端 | Gradio | 快速搭建，适合 Demo 验证 |
| 备选模型 | GLM-4-Flash | 免费，作为降级方案 |

## 关键数据

- 知识库：30 篇飞书帮助文档，130 个文本块（Chunk Size=500）
- 测试题库：15 条，覆盖正常/边界/盲区/乱输入/空输入 5 类
- 评估总分：**4.5/5**（AI 裁判多维度打分）
- 单次推理成本：**¥0.0003**（DeepSeek）或 ¥0（GLM）

## Prompt 迭代记录（3 轮）

| 版本 | 策略 | 正常题正确率 | 空输入处理 | 不知道时的表现 |
|------|------|------------|-----------|---------------|
| v1 简单版 | 基础角色设定 | 50% | 打招呼但附幻觉 | 每次措辞不同，啰嗦 |
| v2 加规则 | 严格规则约束 | 75% | 正常问候 ✅ | 机械复读同一句话 |
| v3 完整版 | 角色+规则+风格+兜底 | 83% | 亲切打招呼 ✅ | 有温度有引导 ✅ |

**v3 胜出**，核心改进：角色设定（"飞小书"）+ 分层规则 + 不知道时的兜底话术 + 口语化风格。

## Bad Case 核心发现

1. **数据源形式不匹配**：飞书帮助中心大量图文教程，纯文本采集丢失信息（图片中的步骤描述无法被检索）
2. **知识库覆盖面**：10 篇真实 + 10 篇合成文档，部分测试题的答案仍不完整（完整性评分 3.7/5）
3. **空输入过度响应**：v3 在空输入时主动推送了修改密码教程，不该揣测用户意图
4. **检索精确度偶有偏差**：个别问题（如审批流程）文档已入库但检索未命中，需排查 Chunk 策略

## 如果产品化——下一步规划

| 维度 | 当前状态 | 产品化方向 |
|------|----------|-----------|
| 数据 | 30 篇静态文档 | 接入飞书帮助中心 RSS，自动更新知识库 |
| 检索 | 单路语义检索 | 混合检索（关键词+语义+重排序） |
| 对话 | 单轮 | 多轮对话 + 上下文理解 |
| 评估 | 离线 AI 裁判打分 | 接入真实用户反馈（👍👎）+ 在线 A/B |
| 部署 | 本地 Gradio | Docker + Nginx + GPU 推理优化 |
| 成本 | ¥150/月(DeepSeek) | 分级路由：简单问题用 GLM 免费模型 |

## 个人反思

这个项目让我深刻理解了 AI PM 的三个核心能力：

1. **数据是第一公民**：RAG 效果的上限由知识库质量决定，不是 Prompt。采集阶段偷的懒会在评估阶段加倍偿还。

2. **PM 需要在成本/质量/延迟之间做权衡**：不是选最好的模型，而是选最合适的。DeepSeek 够用就别上 GPT-4o。

3. **离线指标好 ≠ 产品好**：BLEU/准确率是参考，用户看到的是回答有没有用、读起来顺不顺、知不知道下一步该干嘛。
